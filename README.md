# AWS Portfolio - Flask + ALB + Auto Scaling + RDS

## 概要

AWSを使って、ALB・EC2・RDSで構成されたWebアプリを作成しました。
最初はEC2を1台だけで動かしていましたが、1台構成だと障害時にサービスが止まってしまうため、ALBとAuto Scalingを追加して複数台構成に改善しました。

インフラの基本的な構成だけでなく、「落ちない構成」にすることを意識して作っています。

---

## 構成

![architecture](architecture.png)

構成は以下のようになっています。

* User → ALB → EC2（Flask） → RDS（MySQL）

それぞれの役割は以下です。

* ALB：HTTP(80)でアクセスを受けてEC2に振り分ける
* EC2：Flaskアプリを動かす（port 5000）
* RDS：データ保存用のMySQL

また、EC2はAuto Scalingで複数台起動するようにしています。

---

## この構成にした理由

* EC2単体構成では、1台落ちるとサービスが止まるため
* 実務でよく使われるALB配下構成を経験したかったため
* アプリとDBを分けた構成を理解するため

---

## 工夫した点

* EC2を直接公開せず、ALB経由でのみアクセスできるようにした
* Security Groupで以下の通信だけ許可

  * ALB → EC2（5000）
  * EC2 → RDS（3306）
* Auto Scalingを使い、インスタンスが自動で増減・復旧するようにした
* systemdを使って、EC2起動時にFlaskが自動で起動するようにした

---

## ヘルスチェックについて

ALBのヘルスチェックは `/` に設定しています。

単にポートが開いているかではなく、実際にアプリにアクセスしてレスポンスが返るかを確認することで、アプリケーションが正常に動いているかまでチェックするようにしました。

---

## アプリ機能

* `/` ：投稿一覧表示
* `/add?content=xxx` ：投稿追加

---

## 詰まった点と対応

### ① 503エラー

ALBからEC2にアクセスできず、503エラーになりました。
→ Security Groupの設定を見直して解決しました。

---

### ② ターゲットがunhealthyになる

ヘルスチェックが通らず、ターゲットがunhealthyになりました。
→ Flaskの起動方法やポート設定を見直して解決しました。

---

### ③ AZの設定ミス

ALBとEC2の配置が合っておらず、通信できませんでした。
→ サブネット設定を見直して修正しました。

---

## Auto Scalingの確認

実際にEC2を1台停止して動作を確認しました。

* 停止後、自動で新しいインスタンスが起動
* ALB経由のアクセスは止まらず継続

これにより、障害時でもサービスが継続する構成になっていることを確認できました。
